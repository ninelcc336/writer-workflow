# 正文审查说明

本文件把 `review-draft` 定义成一套可直接执行的流程。

目标只有一个：

- 在人类正式阅读前，先对正文初稿做一轮结构化审查，输出一份可判定是否放行的人审前报告

`review-draft` 的职责是“检查并报告”，不是直接改稿。

## 一、触发条件

满足以下条件后，才允许执行 `review-draft`：

- `draft-chapter` 已完成
- 当前章初稿文件存在
- 当前书的 canon 与 state 可正常读取
- 当前章 plan 文件存在

## 二、禁止前提

有以下任一情况时，不得执行 `review-draft`：

- 当前章初稿不存在
- 当前章 plan 缺失
- state 仍处于明显占位状态
- 正文尚未完整成章

## 三、最小输入

执行 `review-draft` 时，至少需要这些输入：

1. `book/canon/` 基础文件
2. `book/state/*.yaml`
3. 当前章 `plan.md`
4. 当前章 `draft-v1.md` 或最新草稿
5. 当前章 brief
6. 上一章 final 或上一章状态收束

## 四、建议读取顺序

建议 agent 按以下顺序建立审查上下文：

1. `book/canon/premise.md`
2. `book/canon/setting.md`
3. `book/canon/style_rules.md`
4. 当前章 brief
5. 当前章 `plan.md`
6. 当前章最新草稿
7. `book/state/*.yaml`
8. 上一章 final 或上一章状态差异

## 五、审查维度

`review-draft` 至少要检查 4 类问题。

### 1. 计划完成度

检查：

- 是否覆盖计划中的关键节拍
- 是否遗漏必写节点
- 章末落点是否符合计划

### 2. 连续性

检查：

- 角色生死、位置、关系是否冲突
- 势力关系是否无缘无故变化
- 关键资源、装备、能力是否前后打架
- 伏笔推进是否与当前 state 冲突

### 3. 文风与平台风险

检查：

- 模板化副词和套板句
- 机械场景过渡
- 模板化表情或对话配音
- 过长段落
- 模板化章末钩子
- 明显 AI 味表达

### 4. 字数与节奏

检查：

- 是否明显超出预算
- 是否某个节拍异常膨胀
- 是否存在“写了很多但没推进”的问题
- 是否提前透支下一章内容

## 六、问题分级

所有审查结果必须分为 3 类：

### `blocker`

定义：

- 不修不能进入人审的问题

典型情况：

- 关键设定冲突
- 角色状态打架
- 漏写核心节拍
- 章末落点错误
- 明显不可发布级别的结构问题

### `warning`

定义：

- 建议修改，但不一定拦截人审的问题

典型情况：

- 某处节奏偏慢
- 某段略显套路
- 某个小节点表达不够干净

### `note`

定义：

- 仅做提醒，不影响放行

典型情况：

- 可以进一步强化的爽点
- 可选的精简建议
- 风格层面的优化空间

## 七、执行步骤

### 步骤 1：读取计划与草稿

确认本章 plan 和 draft 是匹配版本。

如果 plan 已更新但 draft 仍是旧版本，先标记为 `blocker`。

### 步骤 2：做计划覆盖检查

逐项核对：

- 节拍是否落地
- 必写节点是否出现
- 章末落点是否一致

### 步骤 3：做连续性检查

逐项核对：

- 人物
- 势力
- 伏笔
- 战力/装备/资源

### 步骤 4：做文风风险检查

逐项核对：

- 是否有明显模板化表达
- 是否有平台风险项
- 是否有移动端阅读不友好段落

### 步骤 5：做字数与节奏检查

逐项核对：

- 总字数
- 节拍分布
- 是否拖沓
- 是否越界展开

### 步骤 6：输出审稿报告

建议输出到：

- `book/artifacts/reports/chapter-XXX-draft-review.md`

如果需要，也可附加一份简短结论：

- `放行 / 需修后复审 / 退回重写`

## 八、审稿报告必须包含的内容

一份合格的正文审稿报告至少要有：

1. 本章基础信息
2. 总体结论
3. `blocker`
4. `warning`
5. `note`
6. 对应的修订建议

## 九、与 `humanize-chapter` 的边界

- `humanize-chapter`：偏向主动修订文本
- `review-draft`：偏向检查并产出结论

允许顺序：

1. 先 `review-draft` 再决定是否 `humanize`
2. 先 `humanize` 再做 `review-draft`

但无论顺序如何，正式进入人审前都应有一份 `review-draft` 报告。

## 十、审查阶段禁止事项

在 `review-draft` 阶段，不允许：

- 直接更新 `book/state/*.yaml`
- 把草稿自动升格为 `final.md`
- 用一句“整体还行”替代结构化报告
- 跳过 `blocker` 直接交给人审

## 十一、完成标准

满足以下条件时，可视为 `review-draft` 完成：

- 审稿报告已生成
- 所有问题已分级
- 给出了是否可进入人审的结论

只有在 `blocker` 清零后，才建议进入正式人审。
