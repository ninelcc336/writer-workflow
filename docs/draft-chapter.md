# 正文起草说明

本文件把 `draft-chapter` 定义成一套可直接执行的流程。

目标只有一个：

- 根据已经通过审核的章节计划，产出一份可进入修订和审稿阶段的正文初稿

这里产出的是“初稿”，不是“定稿”。

## 一、触发条件

满足以下条件后，才允许执行 `draft-chapter`：

- `plan-chapter` 已完成
- 用户已明确通过章节计划
- 当前章的 plan 文件存在
- 当前书的 canon 与 state 可正常读取

## 二、禁止前提

有以下任一情况时，不得执行 `draft-chapter`：

- 章节计划尚未通过人审
- 当前章 plan 缺失
- 关键 state 仍不完整或存在明显占位值
- 上下文存在冲突但尚未澄清

## 三、最小输入

执行 `draft-chapter` 时，至少需要这些输入：

1. `book/canon/` 基础文件
2. `book/state/*.yaml`
3. 当前章 `plan.md`
4. 当前章 `brief`（如果存在）
5. 上一章 final 或上一章状态收束

如果是第 1 章，则上一章输入可为空。

## 四、建议读取顺序

建议 agent 按以下顺序建立上下文：

1. `book/canon/premise.md`
2. `book/canon/setting.md`
3. `book/canon/style_rules.md`
4. `book/canon/characters/protagonist.md`
5. 当前卷纲
6. 当前章 brief
7. 当前章 `plan.md`
8. `book/state/*.yaml`
9. 上一章 final 或上一章状态差异

## 五、执行步骤

### 步骤 1：确认计划锁定

先确认当前计划已经通过审核，且以下内容清晰：

- 本章目标
- 节拍拆分
- 字数预算
- 必写节点
- 风险与禁忌
- 章末落点

如果这些内容仍模糊，先返回 `plan-chapter` 修正，不要硬写。

### 步骤 2：建立正文约束清单

起草前，先提炼一份“不可违背清单”，至少包括：

- 必须出现的角色和资源
- 不得冲突的 state 事实
- 必须兑现或推进的伏笔
- 本章不允许越界展开的内容

### 步骤 3：按节拍起草

正文必须按通过审核的节拍顺序推进。

要求：

- 每个节拍都要有明确的功能
- 不得跳过 plan 中的核心节点
- 不得新增会改写大设定的内容
- 不得在中途把章末钩子提前透支完

### 步骤 4：控制字数与节奏

起草时必须持续检查：

- 当前字数是否贴近预算
- 是否有某一节拍异常膨胀
- 是否出现“写了很多，但没推进”的问题

如果发现失控，应在初稿阶段收束，而不是把问题留给修订阶段。

### 步骤 5：完成章末落点

正文结尾必须和计划中的落点一致：

- 钩子推进
- 爽点承诺
- 情绪收束
- 冷收束

不允许把计划中的结尾类型偷偷改掉。

### 步骤 6：输出初稿

建议输出到：

- `book/drafts/chapter-XXX/draft-v1.md`

如果需要保留多个版本，可以继续追加：

- `draft-v2.md`
- `draft-v3.md`

但未经审核，不得覆盖 `final.md`。

## 六、初稿文件必须包含的内容

一份合格的正文初稿至少要满足：

1. 有明确章节标题
2. 主体内容完整成章
3. 已覆盖计划中的关键节拍
4. 具备可供修订的完整结尾

## 七、起草阶段自查项

在进入 `humanize-chapter` 或 `review-draft` 前，至少应自查：

- 是否漏写必写节点
- 是否违反关键 state 约束
- 是否出现重大人设漂移
- 是否偏离本章目标
- 是否明显超出预算
- 是否提前展开了下一章内容

## 八、起草阶段禁止事项

在 `draft-chapter` 阶段，不允许：

- 直接更新 `book/state/*.yaml`
- 把初稿命名为 `final.md`
- 无视 plan 重写整章方向
- 在未说明的情况下新增重大设定
- 跳过修订或审稿直接宣布可发布

## 九、完成标准

满足以下条件时，可视为 `draft-chapter` 完成：

- 正文初稿文件已生成
- 主要节拍已落地
- 可以进入 `humanize-chapter` 或 `review-draft`

完成 `draft-chapter` 不代表本章完成，更不代表可以同步 state。
