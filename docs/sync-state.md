# 状态同步说明

本文件把 `sync-state` 定义成一套可直接执行的流程。

目标只有一个：

- 在正文已经通过人审后，把本章产生的长期有效事实回写到 `book/state/`，并输出一份结构化状态差异报告

`sync-state` 的职责是“同步事实”，不是改正文，也不是补写剧情。

## 一、触发条件

满足以下条件后，才允许执行 `sync-state`：

- 当前章正文已经通过人审
- 当前章存在可视为正式版本的文件
- 当前章 `plan.md` 存在
- 当前章 `draft review` 报告已完成
- 当前书的 `book/state/*.yaml` 可正常读取

## 二、禁止前提

有以下任一情况时，不得执行 `sync-state`：

- 当前章仍处于草稿状态
- 人审尚未明确通过
- 当前章正式版本不存在
- 当前章仍存在未处理的 `blocker`
- state 仍处于模板占位状态

## 三、最小输入

执行 `sync-state` 时，至少需要这些输入：

1. 当前章正式版本
2. 当前章 `plan.md`
3. 当前章 `draft review` 报告
4. `book/state/*.yaml`
5. 上一章 final 或上一章状态差异
6. 当前卷纲和必要 canon 文件

## 四、可以同步什么

只同步“会影响后续写作判断的长期事实”。

典型包括：

- 角色生死、位置、阵营、关系变化
- 势力归属或地盘变化
- 伏笔的新增、推进、回收
- 关键资源的获得、消耗、损毁
- 长期有效的能力、装备、基地模块变化
- 新章节索引条目

## 五、不该同步什么

以下内容不应写入 `state`：

- 修辞性表达
- 短期情绪波动
- 无后续影响的临时动作
- 只属于段落层面的文采优化

原则：

- 事实入 state
- 修辞留在正文

## 六、执行步骤

### 步骤 1：锁定正式版本

先明确本章哪一个文件是人审通过版本。

建议约定：

- `book/drafts/chapter-XXX/final.md`

如果当前章还没有正式版本文件，不应直接同步 state。

### 步骤 2：提取状态变化

从本章正式版本中提取本章的长期有效变化，至少按以下类别扫描：

- 人物
- 势力
- 伏笔
- 能力/装备/资源
- 章节索引

### 步骤 3：与旧 state 对照

逐项比对当前 `book/state/*.yaml`：

- 哪些项目不变
- 哪些项目新增
- 哪些项目更新
- 哪些项目回收或关闭

### 步骤 4：更新对应 state 文件

按类别更新：

- `characters.yaml`
- `factions.yaml`
- `foreshadows.yaml`
- `power_state.yaml`
- `chapter_index.yaml`

要求：

- 不得凭感觉乱补
- 所有变化都应能在正式正文中找到依据

### 步骤 5：生成状态差异报告

建议输出到：

- `book/artifacts/reports/chapter-XXX-state-diff.md`

这份报告至少要回答：

1. 哪些角色变了
2. 哪些伏笔变了
3. 哪些势力关系变了
4. 哪些资源或能力变了
5. 哪些内容确认无变化

### 步骤 6：更新章节索引

在 `chapter_index.yaml` 中补写本章：

- 章节号
- 标题
- 卷归属
- 状态
- 本章摘要
- 必须处理过的 payoff
- 关键状态变化
- 章末钩子

## 七、状态差异报告必须包含的内容

一份合格的状态差异报告至少要有：

1. 本章基础信息
2. 总体结论
3. 角色变化
4. 势力变化
5. 伏笔变化
6. 能力/资源变化
7. 无变化项

## 八、同步阶段禁止事项

在 `sync-state` 阶段，不允许：

- 反向修改正文
- 擅自修正文中不存在的事实
- 用推测补完没有被写出的重大变化
- 跳过状态差异报告

## 九、完成标准

满足以下条件时，可视为 `sync-state` 完成：

- `book/state/*.yaml` 已更新
- 状态差异报告已生成
- 本章章节索引已更新

完成后才能进入 `render-artifacts`。
